# Pipeline performance baseline + first optimisation — 2026-05-22

Captured by `tests/benchmarks/profile_stages.py` against the synthetic 100K
dataset (cached parquet under `tests/benchmarks/data/benchmark_100k/`).
3 runs each, reporting **mean ms**.

## After / before summary

| Framework / config | Before (ms) | After (ms) | Δ |
|---|---:|---:|---:|
| CRR / full IRB | 6 300 | **4 313** | **-1 987 (-31.5 %)** |
| Basel 3.1 / full IRB | not captured pre-change | 5 488 | (~similar Δ expected) |

The change: defer classifier diagnostics (`_emit_model_permission_diagnostics`,
`_collect_beel_on_non_defaulted_warnings`) until after a single
`materialise_barrier` at the end of `ExposureClassifier.classify()`.
Previously these two diagnostic `.collect()` calls each re-executed the full
upstream lazy plan, and CRM's downstream `collect #1` re-executed it a third
time. Now the upstream plan executes once.

### Why this worked

The "Classifier" stage in the baseline reported 888 ms, but the bundle it
returned was still a *lazy* frame. So what was 888 ms actually paying for?
Two diagnostic `.collect()` calls inside `classify()`:

- `_collect_beel_on_non_defaulted_warnings` — counts performing exposures
  with non-zero BEEL.
- The model-permission diagnostic inside
  `_resolve_model_permissions_if_present` — groups failed model matches by
  cause.

Each `.collect()` re-executes the upstream lazy plan independently —
Polars does not cache intermediate results between collects. So the
pipeline was effectively doing three full upstream evaluations:

1. BEEL diagnostic (in classifier)
2. Model permission diagnostic (in classifier)
3. CRM collect #1 (after provisions/CCF/init_ead)

An experiment confirmed this: stubbing both diagnostic collects to
return `[]` saved ~1 100 ms but didn't reduce CRM collect #1
materially — proving they were paying for *upstream re-evaluation*, not
diagnostic-specific work.

The fix is to materialise the classifier output **once** at the end of
`classify()`, then run both diagnostics against the in-memory frame. The
downstream CRM barrier then only pays for its own added transforms
(provisions, CCF, init_ead) — ~250 ms instead of ~2 620 ms.

Future optimisation candidates:
- `collect_all SA+IRB+Slotting` is actual calculator compute (each
  branch's `.with_columns` chains). Reducing this needs work inside the
  per-approach calculators, not the orchestrator.
- `CRM: collect_all (exp+guar lookups)` collects four LazyFrames in
  parallel; one of them (`ri_lf` — rating inheritance) overlaps with
  work the classifier already did. Worth investigating whether it can be
  threaded through instead of re-built.
- Pipeline pre-split `collect #3` is the second materialisation barrier
  in the pipeline. If we can prove the calculator branches don't
  re-execute upstream, this barrier could potentially be removed —
  although the comment at `engine/pipeline.py:639-643` warns the
  guarantee plan is deep enough that CSE alone won't deduplicate.

### Methodology / caveats

- Single machine (Windows laptop), no thread pinning.
- Polars 1.x default thread count (no `POLARS_MAX_THREADS` cap).
- `profile_stages.py` replays the pipeline stages individually outside
  the orchestrator; numbers should be representative of production
  pipeline timing but the breakdown into named barriers is a
  re-implementation, not a tap into the real `stage_timer` instrumentation.
